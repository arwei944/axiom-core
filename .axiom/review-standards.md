# 架构治理代码审查标准

> **最严厉审查标准** — 零容忍任何架构违规、代码异味或潜在风险。

## 1. 架构一致性审查

### 1.1 单一数据源原则
- [ ] 所有架构规则必须来自 `.axiom/architecture.toml`
- [ ] 禁止在任何代码中硬编码架构常量
- [ ] 禁止复制架构数据到多个位置
- [ ] 所有加载点必须使用相同的解析逻辑

### 1.2 分层规则遵守
- [ ] 每个 crate 必须注册到 `[crate-layers]`
- [ ] 依赖方向必须符合 Layer N → Layer >= N
- [ ] 豁免必须有明确理由并记录在案
- [ ] 禁止循环依赖

### 1.3 编译期强制
- [ ] 每个 crate 的 `build.rs` 必须调用架构检查
- [ ] 检查逻辑必须来自单一实现
- [ ] 违规必须导致编译失败（panic）
- [ ] 错误信息必须清晰可操作

## 2. 代码质量审查

### 2.1 代码重复
- [ ] 零重复代码（DRY 原则）
- [ ] 相同逻辑必须提取到共享模块
- [ ] 禁止复制粘贴架构解析代码

### 2.2 错误处理
- [ ] 所有 `Result` 必须正确处理
- [ ] 禁止 `unwrap()` 在生产代码中
- [ ] 错误信息必须包含上下文
- [ ] 失败必须提供修复建议

### 2.3 API 设计
- [ ] 公共 API 必须稳定
- [ ] 禁止暴露内部实现细节
- [ ] 函数签名必须清晰表达意图
- [ ] 必须遵守单一职责原则

### 2.4 性能
- [ ] 编译期检查必须在 1 秒内完成
- [ ] 禁止在 build.rs 中进行 I/O 密集型操作
- [ ] 缓存机制必须正确（OnceLock/Lazy）
- [ ] 内存使用必须最小化

## 3. 安全审查

### 3.1 依赖安全
- [ ] 所有依赖必须经过审计
- [ ] 禁止引入未审计的第三方依赖
- [ ] 依赖版本必须固定（无 `*` 通配符）
- [ ] 必须定期更新依赖

### 3.2 代码注入
- [ ] 禁止在 build.rs 中执行任意代码
- [ ] 所有输入必须验证
- [ ] 路径遍历必须防护
- [ ] 禁止 `include!` 未验证内容

### 3.3 信息泄露
- [ ] 错误信息不能暴露敏感路径
- [ ] 日志不能包含密钥/凭证
- [ ] panic 信息必须安全

## 4. 测试审查

### 4.1 测试覆盖
- [ ] 每个公共函数必须有单元测试
- [ ] 集成测试覆盖主要流程
- [ ] 边界条件必须测试
- [ ] 错误路径必须测试

### 4.2 测试质量
- [ ] 测试必须独立可运行
- [ ] 禁止测试之间的隐式依赖
- [ ] 测试命名必须清晰
- [ ] 断言必须精确

## 5. 文档审查

### 5.1 代码文档
- [ ] 所有公共 API 必须有文档注释
- [ ] 文档必须包含使用示例
- [ ] 复杂逻辑必须注释
- [ ] 禁止过时注释

### 5.2 架构文档
- [ ] 所有规则必须有明确理由
- [ ] 豁免必须记录原因
- [ ] 变更必须有记录
- [ ] 文档必须与实际代码同步

## 6. 可维护性审查

### 6.1 模块化
- [ ] 职责单一，模块高内聚
- [ ] 模块间依赖最小化
- [ ] 禁止 God Module
- [ ] 接口清晰，隐藏实现

### 6.2 可扩展性
- [ ] 新增 crate 流程简单
- [ ] 新增规则不需要修改核心逻辑
- [ ] 配置驱动，非代码驱动
- [ ] 向后兼容保证

## 7. 容错性审查

### 7.1 优雅降级
- [ ] 配置文件缺失必须有明确错误
- [ ] 解析失败必须有修复建议
- [ ] 禁止静默失败
- [ ] 禁止默认继续执行

### 7.2 恢复能力
- [ ] 损坏的 TOML 必须明确报错
- [ ] 缺失的 crate 注册必须明确报错
- [ ] 提供修复指导

## 8. 工具链审查

### 8.1 CLI 设计
- [ ] 命令命名清晰一致
- [ ] 参数设计合理
- [ ] 帮助信息完整
- [ ] 退出码符合标准

### 8.2 输出格式
- [ ] 人类可读格式清晰
- [ ] 机器可读格式（JSON）有效
- [ ] 错误输出到 stderr
- [ ] 正常输出到 stdout

## 9. 兼容性审查

### 9.1 向后兼容
- [ ] 公共 API 签名不变
- [ ] 配置文件格式兼容
- [ ] 行为变更必须有迁移路径
- [ ] 废弃必须提前通知

### 9.2 平台兼容
- [ ] Windows/Linux/macOS 兼容
- [ ] 路径处理跨平台
- [ ] 行尾符处理正确
- [ ] 文件编码统一 UTF-8

## 10. 审查流程

### 10.1 自动化检查
```bash
# 必须全部通过
cargo check --workspace
cargo test --workspace
cargo run -p archcheck -- --validate-architecture
cargo run -p archcheck -- --list-crates
cargo run -p xtask -- gatecheck --strict
```

### 10.2 人工审查清单
- [ ] 所有 TODO 已解决
- [ ] 所有 FIXME 已处理
- [ ] 无调试代码残留
- [ ] 无敏感信息泄露
- [ ] 文档与实际一致

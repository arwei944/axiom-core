# Security Policy

## Reporting Security Issues

如果你发现 Axiom Core 存在安全漏洞，请**不要**通过公开 GitHub Issues 提交。

### 报告流程

1. **发送邮件** 至 `security@example.com`（替换为实际安全邮箱）
2. **邮件主题**: `[SECURITY] 简要描述漏洞类型`
3. **邮件内容**:
   - 漏洞类型（如：注入、权限绕过、拒绝服务等）
   - 复现步骤
   - 影响范围（哪些版本受影响）
   - 建议修复方案（如有）
   - 你的联系方式

### 响应时间

- **24 小时内**：确认收到报告
- **72 小时内**：初步评估和分类
- **7 天内**：提供修复时间表
- **修复后**：协调 CVE 申请和公开披露

---

## 安全更新机制

### 依赖审计

- 每日运行 `cargo audit` 扫描已知漏洞
- 关键依赖升级在 48 小时内完成
- 安全更新发布为 PATCH 版本

### 安全版本

- 标记为 `[CVE-XXXX-XXXX]` 的修复会优先发布
- 仅包含安全修复的版本标记为 `-security` 后缀

## 安全最佳实践

### 输入验证

- 所有外部输入必须验证
- 禁止信任用户提供的配置或数据

### 敏感信息

- 禁止在日志中打印密钥、token、密码
- 使用 `tracing` 时注意脱敏

### 权限最小化

- Cell/Tool 权限必须显式声明
- 默认拒绝未授权访问

### 拒绝服务防护

- 实现速率限制、熔断、超时机制
- 防止资源耗尽攻击

---

## 依赖安全

### 审计流程

1. 新依赖必须通过 `cargo audit` 扫描
2. 加入 `.axiom/architecture.toml` 的 `[audited-deps]`
3. 定期审查传递依赖

### 供应链安全

- 生产环境建议使用 `cargo vendor` 或私有 registry
- 锁定关键依赖版本

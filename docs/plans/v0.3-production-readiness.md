## 1. 背景与目标

### 1.1 为什么是 v0.3？

v0.2 已完成：
- Lens 原语基本实现
- SQLite/文件存储后端
- 约束运行时闭环
- API 稳定化

v0.3 聚焦**从"能用"到"生产可用"的最后一公里**：
- 解决 v0.2 未覆盖的生产场景
- 偿还已知技术债
- 补齐可观测性和运维能力
- 确保确定性语义在测试和生产中一致

### 1.2 核心目标

| 目标 | 度量 | 目标值 |
|------|------|--------|
| 确定性测试友好 | 支持注入时钟 | 100% 核心原语可测 |
| 可观测性 | 遥测导出 | Prometheus + OpenTelemetry |
| 持久化可用 | 默认后端 | SQLite 默认启用 |
| 运行时健壮 | 已知 bug 修复 | 0 未解决 P1/P2 |
| 代码健康 | 复杂度控制 | 单文件 < 600 行 |
| 文档生产化 | 运维文档 | 部署/监控/排障指南 |

### 1.3 不做什么

- **不新增 crate**：18 个 crate 足够，v0.3 只改进现有实现
- **不新增原语**：5 个原语不变，只增强现有原语
- **不改变公开 API 签名**：Stable API 保持不变，只修复实现

## 3. 设计原则

| 原则 | 说明 |
|------|------|
| **向后兼容** | Stable API 不变，只修复实现 |
| **渐进增强** | 新功能通过 feature flag 控制 |
| **零成本抽象** | 新特性无运行时开销（除非显式启用） |
| **测试先行** | 每个任务先写测试，再实现 |
| **文档即代码** | 每个公共 API 必须有文档示例 |

**设计**:

pub trait Clock: Send + Sync {
    fn now_ns(&self) -> u64;

pub struct MockClock {
    now: parking_lot::Mutex<u64>,

impl Clock for MockClock { ... }

## 7. Phase 4: 运行时健壮性

### 目标

修复已知设计细节问题，提升运行时正确性和性能。

### 7.1 tokio::select! 修复

**任务**:

#### Task 4.1: 修复 stop 信号接收

**问题**: `runtime.rs:446` 使用 `_ = &mut rx`，在 tokio 1.x 中不会按预期工作。

**修复方案**:
```rust
// 修复前
_ = &mut rx => { ... }

// 修复后
_ = rx.changed() => { ... }
// 或
Ok(()) = rx.recv().await => { ... }
```

**验收标准**:
- [ ] 修复 stop 信号接收逻辑
- [ ] 添加集成测试验证优雅关闭
- [ ] 压力测试验证关闭时序

**依赖**: 无

### 7.2 Layer 规则审查

**任务**:

#### Task 4.2: 审查并修正 Layer::can_send_to

**当前规则**:
- `Oversight → 所有层` ✅
- `Agent → Agent | Validate` (不能到 Exec) ⚠️
- `Validate → Validate | Exec | Agent` ⚠️
- `Exec → Exec` ✅

**问题**: `Validate → Agent` 和 `Agent → Validate` 不对称，需要确认是否符合业务流。

**验收标准**:
- [ ] 与架构师确认规则
- [ ] 更新 `layer.rs` 注释
- [ ] 更新 `API_BOUNDARY.md`
- [ ] 添加规则测试

**依赖**: 无

### 7.3 SignalEnvelope 序列化优化

**任务**:

**目标**: 在内部总线上使用 bincode 减少 JSON 开销，对外接口保持 JSON。

**方案**:

## 11. 风险与回滚

### 11.1 风险登记册

| ID | 风险 | 概率 | 影响 | 缓解措施 |
|----|------|------|------|----------|
| R-301 | Clock 抽象引入性能开销 | 中 | 低 | 内联优化，基准测试验证 |
| R-302 | Prometheus 依赖增加二进制体积 | 低 | 低 | 通过 feature 控制 |
| R-303 | SQLite 默认启用导致磁盘占用 | 中 | 中 | 文档说明清理策略 |
| R-304 | runtime.rs 拆分引入回归 | 中 | 高 | 小步提交，充分测试 |
| R-305 | Layer 规则变更破坏现有应用 | 低 | 高 | 先确认再修改，提供迁移指南 |

### 11.2 回滚程序

1. **feature flag 回滚**: 所有新功能通过 feature flag 控制，可快速关闭
2. **版本回滚**: 每个 Phase 独立提交，可 revert 单个 Phase
3. **数据迁移回滚**: SQLite migration 保留 down 脚本

## 三、架构约束 — 硬约束清单

### 约束1: Crate层依赖规则（编译期强制）

**规则**: crate at level N may only depend on crates at level >= N

**Crate层级**:
| Level | Crate | 允许依赖 |
|-------|-------|---------|
| 0 | axiom-cli | >=0 (所有crate) |
| 1 | axiom-viz | >=1 |
| 2 | axiom-agent | >=2 |
| 3 | axiom-oversight | >=3 |
| 4 | axiom-runtime | >=4 |
| 5 | axiom-store | >=5 |
| 6 | axiom-macros | >=6 (仅core) |
| 7 | axiom-core | 仅第三方依赖 |

**约束效果**:
- ❌ `axiom-runtime` 不能依赖 `axiom-oversight`（反向依赖）
- ✅ `axiom-oversight` 可以依赖 `axiom-runtime`（正向依赖）
- ✅ `axiom-agent` 可以依赖 `axiom-runtime`、`axiom-core`

**强制机制**: [crates/axiom-core/src/gate.rs](file:///d:/work/trae/axiom-core-project/crates/axiom-core/src/gate.rs#L58-L77) — `verify_dependencies()` 编译期检查

---

### 约束2: 四层架构调用规则（编译期+运行期强制）

**规则**: 只能向下或同层调用

**调用矩阵**:
| 来源 | Oversight | Agent | Validate | Exec |
|------|-----------|-------|----------|------|
| Oversight | ✅ | ✅ | ✅ | ✅ |
| Agent | ❌ | ✅ | ✅ | ❌ |
| Validate | ❌ | ✅ | ✅ | ✅ |
| Exec | ❌ | ❌ | ❌ | ✅ |

**约束效果**:
- ❌ `ExecCell` 无法调用 `AgentCell`（编译失败）
- ❌ `AgentCell` 无法调用 `OversightCell`（编译失败）
- ✅ `AgentCell` 可以调用 `ValidateCell`（合法）

**强制机制**:
- 编译期：[crates/axiom-core/src/sealed.rs](file:///d:/work/trae/axiom-core-project/crates/axiom-core/src/sealed.rs#L41-L49) — `CanSendTo` trait
- 运行期：[crates/axiom-core/src/layer.rs](file:///d:/work/trae/axiom-core-project/crates/axiom-core/src/layer.rs#L31-L38) — `can_send_to()`

## 四、约束对开发的硬性要求

### 新增crate时的约束

**步骤**:
1. 在 `CRATE_LAYERS` 中注册新crate和层级
2. 确保依赖方向正确（只能依赖同层或下层）
3. 使用已审核的第三方依赖
4. 不使用禁止的依赖（如 `async-trait`）

**示例**: 新增 `axiom-rag` crate
```rust
// gate.rs 中注册
pub const CRATE_LAYERS: &[(&str, usize)] = &[
    ("axiom-cli", 0),
    ("axiom-viz", 1),
    ("axiom-agent", 2),
    ("axiom-oversight", 3),
    ("axiom-runtime", 4),
    ("axiom-store", 5),
    ("axiom-macros", 6),
    ("axiom-core", 7),
    ("axiom-rag", 4),   // 新crate，层级4（依赖runtime和core）
];
```

---

### 新增Cell时的约束

**步骤**:
1. 使用 `#[cell(layer = "...")]` 宏标注层归属
2. 通过编译期检查（`CanSendTo` trait bound）
3. 运行时自动遵守层间调用规则

**示例**:
```rust
// ✅ 合法：Agent层Cell
#[cell(layer = "agent")]
impl Cell for MyAgentCell {
    type Message = MyMessage;
    type Layer = AgentLayer;
    
    fn handle(...) {
        ctx.send_to::<ValidateLayer, _>(signal, "validator"); // ✅ 合法
        ctx.send_to::<OversightLayer, _>(signal, "governor"); // ❌ 编译失败
    }
}
```

---

### 新增Signal时的约束

**步骤**:
1. 使用 `#[signal(kind = "...", layer = "...")]` 宏标注
2. 实现 `Signal` trait
3. 实现 `Schema` trait（验证逻辑）

**示例**:
```rust
#[signal(kind = "command", layer = "exec")]
struct MyCommand { ... }

impl Signal for MyCommand { ... }
impl Schema for MyCommand {
    fn validate(&self) -> Result<(), ValidationError> { ... }
}
```

---

### 新增工具时的约束

**步骤**:
1. 实现 `Tool` trait
2. 通过 `ToolRegistry` 注册
3. 添加权限控制（`required_permission`）
4. 每次调用产生Witness记录

**示例**:
```rust
struct MyTool;

#[async_trait]
impl Tool for MyTool {
    fn info(&self) -> ToolInfo {
        ToolInfo {
            name: "my_tool".to_string(),
            description: "...".to_string(),
            parameters: vec![...],
            required_permission: Some("admin".to_string()), // 权限控制
            version: "1.0.0".to_string(),
        }
    }
    
    async fn execute(&self, params: &Value) -> Result<Value, ToolError> { ... }
}
```

## 六、约束违规后果

| 违规类型 | 后果 | 修复方式 |
|----------|------|---------|
| 反向依赖 | 编译失败 | 调整crate层级或依赖关系 |
| 禁止依赖 | 编译失败 | 移除禁止依赖，使用替代方案 |
| 未审核依赖 | 编译失败 | 将依赖添加到 `AUDITED_DEPS` 并审核 |
| 跨层调用 | 编译失败 | 使用合法的调用路径 |
| `unwrap()` / `expect()` | Clippy警告（CI失败） | 使用 `Result` 模式 |
| 缺少文档 | Clippy警告（CI失败） | 添加文档注释 |

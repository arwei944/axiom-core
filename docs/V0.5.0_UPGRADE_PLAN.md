# v0.5.0 Upgrade Task Breakdown

> Target: axiom-core-project
> Baseline: v0.5.0 architecture upgrade direction
> Goal: production-grade distribution, plugin isolation, observability, security hardening, ecosystem readiness

## Legend
- `[ ]` Pending
- `[~]` In progress
- `[x]` Done

---

## P4-1: Distributed Runtime Foundation

- [ ] Define `ClusterNodeId` and `ClusterTopology` types
- [ ] Implement `ClusterTransport` trait with in-memory + gRPC adapters
- [ ] Add node discovery via static config + gossip protocol stub
- [ ] Add `ReplicatedCellKernel` that forwards `send/receive` to remote nodes
- [ ] Add cluster-aware `MessageBus` with topic-based routing
- [ ] Add integration test: 2 nodes, cell on node A sends signal to cell on node B

**Tests**
- **Unit**: `ClusterTopology` resolves node addresses and partitions.
- **Integration**: 2-node cluster routes signal end-to-end and preserves ordering per cell.
- **Chaos**: node B restarts, messages queue on node A, then flush after recovery.

## P4-3: Plugin Hot Update

- [ ] Add `PluginRegistry:: staged_install(id, version, plugin)` API
- [ ] Add `PluginRegistry::promote(id)` to switch traffic to staged plugin
- [ ] Add `PluginRegistry::rollback(id)` to revert to previous version
- [ ] Persist staged plugin metadata to `EventStore`
- [ ] Add CLI commands: `axm plugin stage`, `axm plugin promote`, `axm plugin rollback`
- [ ] Add integration test: stage v2, promote, rollback, verify message handlers swap

**Tests**
- **Upgrade**: stage v2 while v1 handles traffic, promote causes zero dropped messages.
- **Rollback**: after failed promote, rollback restores v1 within one probe interval.
- **Persistence**: crash after stage, restart, staged plugin remains available.
